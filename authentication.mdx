# Authentication & Access Control

The Cloro de Hidalgo Procurement System uses **Supabase Authentication** with a custom role-based access control (RBAC) system to secure data and enforce permissions.

## Authentication Architecture

### Supabase SSR Integration

The application uses Supabase's SSR (Server-Side Rendering) package for seamless authentication across client and server components:

```typescript
// From lib/supabase/server.ts
import { createServerClient } from '@supabase/ssr'
import { cookies } from 'next/headers'

export async function createClient() {
  const cookieStore = await cookies()
  
  return createServerClient(
    process.env.NEXT_PUBLIC_SUPABASE_URL!,
    process.env.NEXT_PUBLIC_SUPABASE_ANON_KEY!,
    {
      cookies: {
        getAll() {
          return cookieStore.getAll()
        },
        setAll(cookiesToSet) {
          // Set cookies for auth persistence
        },
      },
    }
  )
}
```

```typescript
  return NextResponse.redirect(url)
}
```

## User Roles

<Warning>
**Implementation Note**: The TypeScript application code references six roles, but the database schema currently defines only three roles in the PostgreSQL enum (`admin`, `coordinadora`, `consulta`). To use all six roles as intended, the database schema needs migration to add `laboratorio`, `cedis`, and `pendiente` to the `user_role` enum. See [Database Roles](/database/roles) for the current schema definition.
</Warning>

The application code implements six user roles:

```typescript
// From types/index.ts:100-109
export type UserRole = 'admin' | 'coordinadora' | 'laboratorio' | 'cedis' | 'pendiente' | 'consulta'

export interface Profile {
  id: string
  nombre_completo: string | null
  email: string | null
  rol: UserRole
  created_at: string
  updated_at: string
}
```

### Role Definitions

<Steps>
  <Step title="admin">
    **Administrator** - Full system access
    
    - Create, edit, delete requisitions
    - Manage all catalogs (suppliers, products, etc.)
    - Approve/reject user registrations
    - Assign and modify user roles
    - Delete users
    - Access admin panel
    - Edit fecha_confirmada field
  </Step>
  
  <Step title="coordinadora">
    **Coordinator** - Procurement management
    
    - Create, edit, delete requisitions
    - Manage catalogs
    - Approve/reject user registrations
    - Assign roles to new users
    - Access admin panel
    - Edit fecha_confirmada field
    - Cannot delete existing users
  </Step>
  
  <Step title="laboratorio">
    **Laboratory** - Read-only access
    
    - View all requisitions
    - View calendar
    - Filter and search data
    - No create/edit/delete permissions
  </Step>
  
  <Step title="cedis">
    **Distribution Center (CEDIS)** - Read-only access
    
    - View all requisitions
    - View calendar
    - Filter and search data  
    - No create/edit/delete permissions
  </Step>
  
  <Step title="pendiente">
    **Pending** - Awaiting approval
    
    - Automatically assigned on registration
    - No system access until approved
    - Redirected to pending page after signup
    - Must be approved by admin/coordinadora
  </Step>
  
  <Step title="consulta">
    **Consultation (Deprecated)** - Legacy read-only role
    
    - View-only access (same as laboratorio/cedis)
    - No longer assigned to new users
  </Step>
</Steps>

## Role-Based Permissions

The `useAuthRole` hook provides granular permission checks throughout the application:

```typescript
// From lib/hooks/useAuthRole.ts:27-106
export function useAuthRole(): UseAuthRoleReturn {
  const [user, setUser] = useState<User | null>(null)
  const [profile, setProfile] = useState<Profile | null>(null)
  
  // Fetch user and profile on mount
  useEffect(() => {
    async function fetchUserAndProfile() {
      const { data: { user } } = await supabase.auth.getUser()
      setUser(user)
      
      if (user) {
        const { data: profileData } = await supabase
          .from('profiles')
          .select('*')
          .eq('id', user.id)
          .single()
        
        setProfile(profileData)
      }
    }
    
    fetchUserAndProfile()
  }, [])
  
  const role = profile?.rol ?? null
  const isAdmin = role === 'admin'
  const isCoordinadora = role === 'coordinadora'
  const isLaboratorio = role === 'laboratorio'
  const isCedis = role === 'cedis'
  const isPendiente = role === 'pendiente'
  const isConsulta = role === 'consulta'
  
  // Computed permissions
  const canCreate = isAdmin || isCoordinadora
  const canEdit = isAdmin || isCoordinadora
  const canDelete = isAdmin || isCoordinadora
  const canViewOnly = isLaboratorio || isCedis || isConsulta
  const canManageUsers = isAdmin || isCoordinadora
  const canAccessAdmin = isAdmin || isCoordinadora
  
  return {
    user,
    profile,
    role,
    isAdmin,
    isCoordinadora,
    isLaboratorio,
    isCedis,
    isPendiente,
    isConsulta,
    canCreate,
    canEdit,
    canDelete,
    canViewOnly,
    canManageUsers,
    canAccessAdmin,
    loading,
  }
}
```

## User Registration Flow

### 1. User Signs Up

New users register through the self-service signup form:

```typescript
// From lib/actions/auth.ts:23-42
export async function signUp(formData: { 
  email: string; 
  password: string; 
  nombre_completo: string 
}) {
  const supabase = await createClient()
  
  const { error } = await supabase.auth.signUp({
    email: formData.email,
    password: formData.password,
    options: {
      data: {
        nombre_completo: formData.nombre_completo,
      }
    }
  })
  
  if (error) {
    return { error: error.message }
  }
  
  revalidatePath('/', 'layout')
  redirect('/dashboard/pendiente')
}
```

<Note>
  The user's `nombre_completo` is stored in Supabase's `auth.users.raw_user_meta_data` field, then copied to the `profiles` table by a database trigger.
</Note>

### 2. Profile Created with Pending Role

A Supabase database trigger automatically creates a profile with `rol = 'pendiente'`:

```sql
-- Database trigger (conceptual representation)
CREATE TRIGGER on_auth_user_created
  AFTER INSERT ON auth.users
  FOR EACH ROW
  EXECUTE FUNCTION public.handle_new_user();

-- Function creates profile with pendiente role
CREATE FUNCTION public.handle_new_user()
RETURNS TRIGGER AS $$
BEGIN
  INSERT INTO public.profiles (id, email, nombre_completo, rol)
  VALUES (
    NEW.id,
    NEW.email,
    NEW.raw_user_meta_data->>'nombre_completo',
    'pendiente'
  );
  RETURN NEW;
END;
$$ LANGUAGE plpgsql SECURITY DEFINER;
```

### 5. Or Admin Rejects User

```typescript
// From lib/actions/users.ts:60-67
export async function rejectUser(userId: string) {
  // Need admin client to delete auth user
  const adminClient = await getAdminClient()
  const { error } = await adminClient.auth.admin.deleteUser(userId)
  
  revalidatePath('/dashboard/admin')
  return { error: error?.message }
}
```

<Warning>
  Rejecting a user permanently deletes their account from Supabase Auth. They must re-register if they want to request access again.
</Warning>

## Database Security (RLS)

<Note>
  The application relies on **Supabase Row Level Security (RLS)** policies to enforce data access at the database level. Even if client-side checks are bypassed, the database will reject unauthorized operations.
</Note>

Typical RLS policies for this system:

```sql
-- Example: profiles table RLS
ALTER TABLE profiles ENABLE ROW LEVEL SECURITY;

-- All authenticated users can read profiles
CREATE POLICY "Users can view profiles"
  ON profiles FOR SELECT
  USING (auth.role() = 'authenticated');

-- Users can update their own profile
CREATE POLICY "Users can update own profile"
  ON profiles FOR UPDATE
  USING (auth.uid() = id);

-- Only admins can delete profiles
CREATE POLICY "Admins can delete profiles"
  ON profiles FOR DELETE
  USING (
    EXISTS (
      SELECT 1 FROM profiles
      WHERE id = auth.uid()
      AND rol = 'admin'
    )
  );
```

## Best Practices

<Steps>
  <Step title="Always check permissions server-side">
    Don't rely solely on client-side permission checks. Use RLS policies and server action validation.
  </Step>
  
  <Step title="Use the useAuthRole hook consistently">
    Import and use the hook in every component that needs permission checks:
    
    ```typescript
    import { useAuthRole } from '@/lib/hooks/useAuthRole'
    
    const { canEdit, loading } = useAuthRole()
    
    if (loading) return <Spinner />
    if (!canEdit) return <AccessDenied />
    ```
  </Step>
  
  <Step title="Protect admin routes">
    Always verify `canAccessAdmin` on admin pages:
    
    ```typescript
    // From app/dashboard/admin/page.tsx:356-360
    useEffect(() => {
      if (!loading && !canAccessAdmin) {
        router.replace('/dashboard/calendar')
      }
    }, [loading, canAccessAdmin, router])
    ```
  </Step>
  
  <Step title="Assign appropriate roles">
    - Use **laboratorio** or **cedis** for view-only users
    - Reserve **coordinadora** for procurement team members
    - Limit **admin** to IT staff or senior management
  </Step>
</Steps>

## Troubleshooting

### User Can't Access After Approval

**Symptoms:** User logs in but is redirected to pending page or has no access.

**Solution:**
- Verify the user's role in Supabase dashboard
- Check that role was updated from 'pendiente' to a valid role
- Have user log out and log back in to refresh session

### Permission Denied Errors

**Symptoms:** User sees "Access Denied" or actions fail silently.

**Solution:**
- Verify user's current role with `useAuthRole`
- Check RLS policies in Supabase
- Ensure service role key is set for admin operations

### Can't Delete Users

**Symptoms:** Delete user button doesn't work or throws error.

**Solution:**
- Verify `SUPABASE_SERVICE_ROLE_KEY` is set in environment variables
- Check that current user is admin (not coordinadora)
- Review server logs for specific error messages

## Security Considerations

<Warning>
  **Security Checklist:**
  
  ✅ Service role key is only used server-side (Server Actions, API routes)  
  ✅ RLS policies are enabled on all tables  
  ✅ Client-side permission checks are supplemented with server-side validation  
  ✅ Middleware protects all dashboard routes  
  ✅ Sensitive operations (delete user, approve user) require elevated roles  
  ✅ Password requirements enforce minimum 6 characters  
  ✅ Session cookies are HTTP-only and secure  
</Warning>
